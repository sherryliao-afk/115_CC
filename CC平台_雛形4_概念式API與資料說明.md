# CC 平台｜儀表板與模型分享概念式 API／資料說明

文件日期：2026/07/24  
參考雛形：`分享機制_雛形_4_展示回饋修正版.html`  
文件用途：既有 CC 系統功能擴充時，供工程師理解畫面需要的資料、可能的 API 行為及資料模型影響。

#### 本次不納入失敗率指標及相關趨勢圖的 API 討論。

## 一、既有系統與 Portal 的關係

- 登入、登出及帳號管理沿用總平台 Portal。
- CC 平台不需要另外建立一套登入帳號或密碼。
- 正式系統依登入者的 Portal 帳號／既有 CC 角色顯示對應功能。
- 雛形中的角色切換只供展示，不代表正式系統要提供切換功能。

目前測試帳號：

| 平台角色 | Portal 帳號 |
| --- | --- |
| 管理員 | `adminnchc@narlabs.org.tw` |
| 一般使用者 | `user01@narlabs.org.tw` |

需要先請工程師確認：

1. 現有系統登入後如何取得 Portal 使用者資訊。
2. 現有 User 資料是否已有 Portal User ID、Account ID 或其他唯一識別值。
3. 管理員／一般使用者角色是由 Portal 傳入，還是由 CC 系統自行管理。

Email 可作顯示及搜尋，但模型授權、群組成員及用量資料應盡量關聯既有 `UserId`，不要只使用 Email。

## 二、兩種角色層級

本功能同時存在兩種不同層級的角色，不能混為同一個欄位。

### 2.1 平台角色

| 平台角色 | 控制範圍 |
| --- | --- |
| 管理員 | 管理員儀表板及全平台資料 |
| 一般使用者 | 個人儀表板及自己的平台資料 |

### 2.2 模型分享角色

依雛形 4，模型分享角色已收斂為三種：

| 模型功能 | 檢視者 | 下載者 | 擁有者 |
| --- | :---: | :---: | :---: |
| 檢視模型 | ✓ | ✓ | ✓ |
| 下載成果 | — | ✓ | ✓ |
| 編輯模型資訊 | — | — | ✓ |
| 新增分享對象 | — | — | ✓ |
| 移除他人權限 | — | — | ✓ |
| 刪除模型 | — | — | ✓ |

例如，一名 Portal 一般使用者可以是模型 A 的擁有者，也可以只是模型 B 的檢視者。

### 2.3 雛形 4 的擁有者規則需確認

雛形的分享下拉選單允許把新的使用者或群組設成「擁有者」，這代表畫面概念上可能允許多位擁有者。

但雛形中擁有者不提供角色調整及移除按鈕。因此開發前需確認：

1. 一個模型是否允許多位擁有者。
2. 分享給群組時是否可以把整個群組設為擁有者。
3. 新增的擁有者是否與模型建立者權限相同。
4. 擁有者是否永遠不能移除，或只能由主要擁有者／平台管理員處理。
5. 既有模型若已有 `OwnerUserId`，新增的擁有者要視為共同擁有者，還是代表所有權移轉。

這一項會直接影響資料模型，不應由工程師依畫面自行猜測。

## 三、個人儀表板可能需要的 API

概念上可由一支 API 取得登入者的個人儀表板：

```text
GetMyDashboard()
```

前端不需傳入任意 `userId`；後端直接使用目前登入者身分查詢，避免查到其他人的資料。

### 3.1 畫面資料

| 區塊 | 需要的資料 |
| --- | --- |
| 我的概況 | 待提交、執行中、已完成、失敗任務數 |
| 執行中任務 | 任務名稱、專案／資料夾名稱、影像數、建立時間、兩階段狀態及進度 |
| 儲存用量 | 影像用量、影像數、相簿數、模型成果用量、模型數、任務成果數 |
| 我的方案 | 方案名稱、月費、說明及方案項目 |
| 用量與配額 | 總儲存空間、建模數量、相簿數量的已使用量及上限 |

### 3.2 概念回傳

- 任務階段至少要能表達等待中、執行中、完成及失敗。

## 四、管理員儀表板可能需要的 API

概念上可由一支 API 取得全平台儀表板：

```text
GetAdminDashboard()
```

只有平台管理員可以取得；一般使用者不可因為知道 API 名稱就取得全平台資料。

### 4.1 畫面資料

| 區塊／圖表 | 需要的資料 |
| --- | --- |
| 平台概況 | 任務總數、本月新增任務數、執行中任務數、本月完成任務數 |
| 平台使用量 | 全平台影像儲存量、當月活躍使用者數 |
| 月任務量趨勢 | 近 6 個月的完成任務數及失敗任務數 |
| 任務狀態分布 | 待提交、執行中、已完成、失敗任務數 |
| 建模時長分布 | 已完成任務的影像數及處理時間 |
| 任務監控 | 任務名稱、使用者、開始時間、兩階段進度、影像數、已耗時 |

任務監控在雛形中可依下列方式切換排序：

- 最新任務。
- 使用影像數。
- 任務耗時。

概念上可由同一支 API 帶入排序條件：

```text
GetPlatformTasks(sortBy = latest | imageCount | duration)
```

「活躍使用者」依目前既有說明，可定義為本月有任務紀錄的使用者數；若需求方要改成登入次數或最近登入時間，需另外確認。

### 4.3 API 是否拆分

不需要強制每張圖一支 API，可依既有系統狀況選擇：

| 作法 | 適用情況 |
| --- | --- |
| 個人／管理員各一支聚合 API | 資料量不大，希望前端整合單純 |
| KPI、歷史圖表、任務監控分開 | 各區塊查詢成本或更新頻率不同 |
| 建模時長分布獨立 API | 已完成任務很多，需要限制期間或點數 |

若任務監控要定時更新，不代表近 6 個月的圖表也必須每次重新統計。歷史資料可使用既有快取、彙總查詢或較低更新頻率。

## 五、模型分享可能需要的 API

### 5.1 模型畫面摘要

雛形 4 顯示：

- 圖片數量。
- 外部帳號數量。
- 最近權限異動日期。
- 目前具有存取權的使用者／群組。
- 權限異動紀錄。

概念 API：

```text
GetModelSharing(modelId)
```

概念回傳：

```json
{
  "modelId": "model-id",
  "imageCount": 970,
  "externalAccountCount": 1,
  "lastPermissionChangedAt": "2026-07-06T10:20:00+08:00",
  "permissions": [
    {
      "permissionId": "permission-id-1",
      "targetType": "user",
      "targetId": "user-id",
      "displayName": "John Doe",
      "email": "john.doe@example.com",
      "role": "owner",
      "status": "internal"
    },
    {
      "permissionId": "permission-id-2",
      "targetType": "group",
      "targetId": "group-id",
      "displayName": "工程團隊",
      "memberCount": 4,
      "role": "downloader",
      "status": "group"
    },
    {
      "permissionId": "permission-id-3",
      "targetType": "user",
      "targetId": "external-user-id",
      "displayName": "甲方審查窗口",
      "email": "reviewer@example.com",
      "role": "viewer",
      "status": "pendingApproval"
    }
  ],
  "permissionAudits": []
}
```

### 5.2 分享操作

| 畫面行為 | 概念 API |
| --- | --- |
| 搜尋可分享的使用者 | `SearchUsers(keyword)` |
| 搜尋可分享的群組 | `SearchGroups(keyword)` |
| 新增使用者或群組授權 | `AddModelPermission(modelId, targetType, targetId, role)` |
| 將檢視者改為下載者，或反向調整 | `UpdateModelPermission(permissionId, role)` |
| 移除檢視者／下載者 | `RemoveModelPermission(permissionId)` |
| 查看權限異動紀錄 | `GetModelPermissionAudits(modelId)` |

依雛形，擁有者不顯示修改或移除操作。後端仍必須驗證，不可只靠前端隱藏按鈕。

目前畫面以指定帳號及群組分享為主，不包含公開連結 API。

### 5.3 外部帳號

雛形將非內部帳號標示為「外部待審核」，但仍需確認：

1. 系統如何判斷內部／外部帳號，不能直接照雛形的示意網域寫死。
2. 外部帳號是否已存在於 Portal／CC User 資料。
3. 若不存在，是建立邀請紀錄、暫存帳號，還是必須先完成 Portal 帳號申請。
4. 待審核期間是否完全不可讀取模型。
5. 由平台管理員或模型擁有者審核。
6. 是否寄送邀請通知。

在流程未確認前，不建議直接把任意 Email 寫入正式模型授權資料。

## 六、群組管理可能需要的 API

### 6.1 畫面資料

群組畫面需要：

- 群組清單及各群組成員數。
- 群組名稱、建立者、建立日期。
- 已共用模型數。
- 群組來源，目前雛形顯示「平台自建」。
- 群組成員、Email、加入者及加入日期。

### 6.2 操作

| 畫面行為 | 概念 API |
| --- | --- |
| 查詢可見群組 | `GetGroups()` |
| 搜尋群組名稱 | 可由 `GetGroups(keyword)` 或前端篩選 |
| 建立群組 | `CreateGroup(name)` |
| 查詢群組明細 | `GetGroup(groupId)` |
| 依姓名或 Email 搜尋既有使用者 | `SearchUsers(keyword)` |
| 新增群組成員 | `AddGroupMember(groupId, userId)` |
| 移除群組成員 | `RemoveGroupMember(groupId, userId)` |
| 刪除群組 | `DeleteGroup(groupId)` |

雛形的輸入提示雖提到「使用者名稱、Email 或群組名稱」，但實際呈現的是從聯絡人目錄挑選使用者，沒有清楚呈現群組內再加入另一個群組。因此本次不先假設支援巢狀群組。

### 6.3 刪除群組

- 群組明細需回傳已共用模型數，供刪除確認訊息顯示。
- 刪除群組後，該群組對各模型的群組授權會失效。
- 成員若對同一模型另有個人授權，不應被一起移除。
- 後端執行刪除時仍需重新確認影響資料，不能只相信前端顯示的數量。

## 七、User table 可能需要確認的欄位

因現有系統已存在，先請工程師檢查既有 User 資料，不先要求重建。

| 可能需要的欄位 | 目的 |
| --- | --- |
| Portal 唯一帳號 ID | 穩定對應 Portal 帳號 |
| Email | 顯示、搜尋及通知 |
| 顯示名稱 | 畫面顯示 |
| 平台角色 | 管理員／一般使用者；若 Portal 已提供，可能只需沿用或同步 |
| 帳號狀態 | 啟用、停用、待審核等 |
| 最後同步時間 | 了解 Portal 帳號資料何時更新 |

下列資料不建議直接增加在 User table：

- 某個模型的分享角色。
- 所屬群組清單。
- 每月建模次數歷程。
- 儲存用量歷程。
- 權限異動歷程。

這些資料會隨模型、群組或期間產生多筆，應以關聯資料或既有交易資料管理。

## 八、模型分享可能需要另外管理的資料

資料模型至少要能表達：

```text
模型 + 使用者 + 模型角色
模型 + 群組 + 模型角色
群組 + 使用者
模型權限異動紀錄
```

概念上可能需要：

- 模型與使用者的授權關聯。
- 模型與群組的授權關聯。
- 平台自建權限群組。
- 群組成員關聯。
- 模型權限異動紀錄。
- 外部帳號邀請／審核資料；僅在確認納入本期時需要。

所有關聯應使用既有 `ModelId`、`UserId`、`GroupId`，不要只存 Email、顯示名稱或群組名稱。

### 8.1 權限異動紀錄

至少記錄：

- 哪一個模型。
- 誰執行。
- 新增、變更或移除哪個使用者／群組。
- 異動前角色。
- 異動後角色。
- 發生時間。

概念 API：

```text
GetModelPermissionAudits(modelId)
```

## 九、使用者配額與用量

### 9.1 配額

配額通常應與 User 基本資料分開管理，可能包含：

- 所屬方案。
- 總儲存空間上限。
- 每期建模次數上限。
- 相簿數量上限。
- 生效日期。

原因是配額可能被管理員調整，也可能隨方案或期間改變。

### 9.2 用量

先確認既有系統能否由原有資料計算：

| 用量 | 可能取自既有資料 |
| --- | --- |
| 影像儲存量 | 影像／檔案紀錄的檔案大小 |
| 模型成果用量 | 模型產出檔案大小 |
| 影像數 | 影像資料筆數 |
| 相簿數 | 相簿資料 |
| 模型數 | 模型資料 |
| 任務成果數 | 已產生成果的任務資料 |
| 本期建模次數 | 任務／工作紀錄 |

如果既有資料可以正確計算，它們應作為真實來源，不需再維護一份容易不同步的使用量欄位。

若儀表板查詢太慢，可另做「用量彙總／快照」供讀取，但仍應能由原始資料重算。

建模次數需由需求方確認計量時點：

- 建立任務時。
- 送入佇列時。
- 開始建模時。
- 建模成功時。
- 失敗後重新執行是否再算一次。

## 十、直接跟工程師確認的問題

1. 現有 User 資料是否已有 Portal User ID？登入後如何取得平台角色？
2. 現有任務是否已有建立者、狀態、兩階段進度及開始／完成時間？
3. 影像與模型成果是否有保存檔案大小，可否依使用者加總？
4. 現有模型如何記錄建立者或擁有者？
5. 是否已有設備授權、帳號群組或權限異動紀錄可沿用？
6. 儀表板適合回一包資料，還是拆成 KPI、歷史圖表及任務監控？
7. 歷史圖表是否需要彙總資料，避免每次重新掃描所有任務？
8. 模型是否允許多位擁有者及群組擁有者？
9. 同一使用者同時有個人及群組授權時，有效權限如何合併？
10. 外部帳號待審核流程及審核人是誰？
11. 群組由誰建立、哪些人可看及維護？
12. 權限異動紀錄需要保存多久？

## 十一、工程師應理解的核心

- 不重做 Portal 登入，只沿用既有身分。
- 一般使用者與管理員的主要差別是儀表板資料範圍。
- 儀表板是把既有任務、檔案、模型及帳號資料整理成畫面需要的統計。
- 模型角色依雛形 4 為檢視者、下載者、擁有者。
- 模型共享需要模型對使用者／群組的授權關聯，不是在 User table 放一個固定模型角色。
- 配額通常需要獨立管理；用量優先由既有交易資料計算。
- API 名稱及拆分方式由工程師依現有系統慣例決定，本文只表達資料、行為及權限邊界。
