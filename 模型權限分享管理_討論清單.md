# CC／Portal 模型權限分享管理｜會議確認事項整理

會議對象：甲方
會議目的：確認下一期「模型權限分享管理」的既有機制、預計做法與待決策範圍，讓雛型與後續規格書可以收斂。
文件用途：作為會議前溝通、需求收斂、雛型調整與後續規格書撰寫依據。

## 一、會議定調

依既有平台成果與後續規格方向，CC／Portal 已具備帳號管理、角色管理、群組管理、三維建模、模型預覽與下載等基礎，因此下一期不重新討論帳號系統與三維建模流程，而是以既有機制為基礎，補足「模型產出後的分享、授權、管理與異動紀錄」。

第一期建議採模型層級權限作為最小可交付範圍，分享方式先限制為指定帳號與指定群組，不開放公開連結；群組先採平台內自建群組，不串接 AD、LDAP 或其他既有組織群組；角色先採「擁有者、管理者、檢視者」三層，下載權限先獨立為模型設定，不另設下載者角色。

本次需請甲方決策的範圍，主要為外部帳號是否納入第一期、外部帳號審核責任、下載權限是否需與檢視權限分開、群組建立與管理責任、權限異動紀錄保存期限，以及擁有者移轉是否納入第一期。

## 四、建議刪掉或改成一句確認的項目

原本會議稿中有些問題會讓討論發散，甚至讓會議被拉回帳號整合、組織同步或進階資安控管。建議改成以下處理方式。

| 原本問題 | 建議處理 | 原因 |
| --- | --- | --- |
| CC 平台目前帳號來源是自建、LDAP、AD、OAuth，或其他機制？ | 不要當主問題問。改成：「第一期先沿用既有 CC／Portal 帳號，不新增身分來源串接。」 | 這會把會議導向帳號整合，不利收斂。 |
| 群組管理是否串接既有組織群組？ | 改成：「第一期先採平台內群組，不串接 AD／LDAP 群組。」 | 串組織群組會牽涉帳號同步、停用同步、權限繼承。 |
| 是否支援公開連結？ | 建議直接定調：「第一期不做公開連結。」只問是否有驗收展示必須使用。 | 公開連結牽涉資安、到期、外流與稽核。 |
| 是否要資料夾／專案層級權限？ | 建議直接定調：「第一期只做模型層級。」 | 權限繼承會大幅增加規格複雜度。 |
| 是否需要稽核報表匯出？ | 改成：「先留異動紀錄，報表匯出列後續擴充。」 | 稽核資料要留，但報表不一定第一期要做。 |
| 是否需要複雜通知規則？ | 改成：「第一期只做新增分享與外部帳號審核通知。」 | 通知太多會造成使用者干擾，也會增加開發成本。 |
| 是否需要下載浮水印／防外流？ | 改成：「第一期先做下載權限控管，浮水印與防外流列後續。」 | 這屬進階資安控管，不該混在第一版分享管理。 |

## 五、仍需甲方決策的項目

以下才是本次會議真正需要甲方決策的重點。

| 決策項目 | 為什麼要問 | 建議預設立場 |
| --- | --- | --- |
| 外部帳號是否納入第一期 | 會影響註冊、審核、通知與停用 | 第一版可先做「邀請後待審核」，不開放自助註冊。 |
| 外部帳號由誰審核 | 涉及責任歸屬 | 預設由平台管理員審核。 |
| 模型角色要幾級 | 會影響資料庫、畫面、驗收 | 第一版建議三層：擁有者、管理者、檢視者。 |
| 下載權是否獨立控管 | 模型預覽與下載風險不同 | 建議下載不要做成角色，改成模型設定。 |
| 群組由誰建立 | 會影響管理權限 | 建議平台管理員與模型管理者可建立，但群組作用範圍要分清楚。 |
| 群組是個人群組還是平台共用群組 | 會影響可見範圍與維護責任 | 第一版建議「建立者可管理，管理員可全域管理」。 |
| 移除權限是否通知 | 會影響通知量 | 第一版可不主動通知被移除者，只記錄異動。 |
| 權限異動紀錄保存多久 | 涉及資安與維運 | 建議先請甲方給保存年限；未定則先用系統預設保存。 |
| 擁有者是否可移轉 | 會影響離職、交接 | 第一版可由管理員移轉，不開放一般使用者自行移轉。 |

## 六、第一版建議納入範圍

第一版建議以「可驗收、可開發、可收斂」為原則，不把所有進階權限一次做完。

### 6.1 建議納入下一期

| 項目 | 說明 |
| --- | --- |
| 模型層級指定使用者分享 | 可針對單一模型指定可存取帳號。 |
| 平台內自建群組分享 | 以平台內群組作為批次授權工具。 |
| 基本角色 | 第一版採擁有者、管理者、檢視者。 |
| 下載權限設定 | 下載權與檢視權分開，由模型設定控管。 |
| 權限異動紀錄 | 記錄新增、移除、角色變更、操作者與時間。 |
| 外部帳號待審核狀態 | 若甲方同意納入外部帳號，採邀請後待審核。 |
| 權限移除提示 | 移除帳號或群組授權前，提示影響範圍。 |
| 群組刪除前影響提示 | 刪除群組前顯示被哪些模型引用。 |

### 6.2 建議暫緩

| 項目 | 暫緩原因 |
| --- | --- |
| 公開連結分享 | 涉及資安、有效期限、外流與稽核。 |
| 資料夾／專案層級繼承權限 | 權限繼承規則複雜，容易影響第一版驗收。 |
| 串接 LDAP／AD 群組 | 牽涉帳號來源、組織同步與停用同步。 |
| 複雜通知規則 | 容易造成通知過量，也會增加開發成本。 |
| 稽核報表匯出 | 先留紀錄即可，匯出列後續擴充。 |
| 下載浮水印或進階防外流 | 屬進階資安控管，不建議放入第一版。 |
| 一般使用者自行移轉擁有者 | 涉及交接、離職與責任歸屬，建議由管理員執行。 |

## 七、會議時建議直接問甲方的 8 題

原本問題較分散，建議收斂成以下 8 題即可。

1. 下一期是否同意只做「模型層級權限」，資料夾／專案層級繼承權限延後？
2. 分享方式是否同意第一期僅支援「指定帳號／指定群組」，不開放公開連結？
3. 外部帳號是否納入第一期？若納入，是否採邀請後審核制？
4. 外部帳號審核責任由誰負責：平台管理員、模型擁有者，或指定專案管理者？
5. 模型角色是否同意採「擁有者／管理者／檢視者」三層？
6. 下載權限是否需與檢視權限分開控管？
7. 群組是否先採平台內自建群組，不串接 AD／LDAP／既有組織群組？
8. 權限異動紀錄是否為驗收必要項目？保存期限由甲方指定，或先採系統預設？

## 八、建議會議結論格式

每個議題建議用同一種格式收斂，避免會後無法轉成規格。

| 欄位 | 填寫方式 |
| --- | --- |
| 決策 | 採用／不採用／延後／需甲方補資料 |
| 負責方 | 甲方／GIS 中心／雙方確認 |
| 是否進入下一期 | 是／否 |
| 限制條件 | 資安政策、既有帳號系統、驗收範圍、維運成本 |
| 對雛型影響 | 需修改畫面／只改文字／不影響 |
| 對規格書影響 | 新增需求／調整需求／列後續擴充 |

## 九、建議會議結論文字

以下文字可直接作為會議紀錄或會議結論草稿。

下一期模型權限分享管理，原則上沿用既有 CC／Portal 帳號、角色與群組管理基礎，第一期以模型層級授權為範圍，支援指定帳號與平台內群組分享，不開放公開連結，不納入資料夾／專案層級繼承；角色先採擁有者、管理者、檢視者，下載權限另以模型設定控管，權限異動需留存紀錄。外部帳號、審核責任、權限紀錄保存期限與擁有者移轉機制，待甲方於本次會議確認後納入正式需求規格。
